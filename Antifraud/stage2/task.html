Description

Enterprise applications like an anti-fraud system are always used by different types of users having various access levels. It is not necessary that every user can access every part of the system. It becomes more important in the financial systems.

In this stage, a couple of rest endpoints are added to the system that enables user management. For the sake of simplicity, only two types of the user are considered. In real systems, however, there is a wider variety of roles and different access levels.

An anti-fraud system consists of two types of users:

  • Normal users who are able to ask whether a transaction is allowed
  • Admin users that can in addition to the query for a transaction, can change the main settings of the antifraud system e.g. add a  stolen card number, add a prohibited IP address, etc. Furthermore, admin users can add/update/remove other users.

 

Objectives

Assume data class User  with the following properties:

  • name: Complete the name and family of the user.
  • username: username of the user.
  • reole: the role of the user.

Add the following rest-endpoint

  1. An endpoint enables admin user(s) to add a user POST /antifraud/user enabling add new user returning HTTP status code
    • HTTP 201 Created response status code if the user has been successfully added.
    • HTTP 409 Conflict response status code if the username is duplicate.
    • HTTP 422 Unprocessable Entity if any of the three properties is missing.
  2. An endpoint enables admin user(s) to see a list of all users GET /antifraud/user  returning HTTP status code 200 OK.
  3. An endpoint enables the admin user(s) to delete a user DELETE /antifraud/user/{username} returning HTTP status code
    • HTTP 404 Not Found if such a username has not been added before.
    • HTTP 200 OK if the operation has been completed succesfully.
